# Refresh /login and /account to match DS canvas

Labels: next, area:web
Relates: #72, #91, #97

## Goal
Canvas `2a`/`2g` show a simplified login screen (Google/Apple/anonymous-browse buttons, social-proof strip, no email form) and an account page with a "Who are you scouting for?" chip card, notification toggles, and a demoted danger zone. The actual `web/src/pages/LoginPage.tsx` and `web/src/pages/AccountPage.tsx` are both substantially further along than the canvas assumed in places, and missing pieces in others — read the discrepancies below before implementing, since some "add this" items in the canvas note are already partially built.

## Changes — Login (`web/src/pages/LoginPage.tsx`)
1. **Discrepancy to note in the PR:** the canvas `2a` mock shows only three buttons (Google, Apple, "Just browse the map for now") plus a mascot and social-proof strip — no email/password form. The actual `LoginPage.tsx` is a combined admin+public login page with a full email/password sign-in/sign-up form (`handleSubmit`, `mode` state) in addition to the Google button, gated by `isAdminSite`. Decide explicitly whether email/password stays as a secondary option below the primary buttons (recommended — don't remove existing auth paths without confirming they're unused) or gets hidden behind a "more options" disclosure; do not silently delete the email/password path without flagging it, since it may be relied on by synthetic-user test flows (see `docs/SYNTHETIC_USERS.md`).
2. Add the anonymous-browse entry: a ghost-style button "Just browse the map for now" that navigates to `/map` without requiring auth. `web/src/lib/routeAccess.ts` already treats `/map` as a public route (`isPublicRoute`), so this is primarily a navigation link/button, not new access-control logic — reuse `useNavigate()`/`<Link to="/map">`.
3. Apple sign-in: `api/ttf_api/apple_sign_in.py` already implements server-side Apple token exchange/revocation. Add an "Continue with Apple" button to `LoginPage.tsx` wired to whatever client-side Apple auth flow the existing `useAuth()`/`signInWithGoogle()`-equivalent pattern uses — check `web/src/auth/` for whether an Apple sign-in method already exists on the auth context; if not, this issue should add the button as a placeholder gated behind a feature flag (iOS Phase 3 ships the full flow per `docs/MODERNIZATION.md` "Platforms") rather than half-wiring it.
4. Social-proof strip: add a footer element like "Parents near you logged N kid-food speeds this week" replacing empty space below the auth buttons — source the count from an existing stats endpoint if one exists (check `api/ttf_api/routers/` for an admin overview stat that could be reused publicly, e.g. the `CONTRIBUTIONS · 7D` figure referenced in canvas `5a`); if no public-safe endpoint exists, scope this down to a static/config-driven number rather than exposing an admin-only aggregate.

## Changes — Account (`web/src/pages/AccountPage.tsx`)
5. **Discrepancy to note in the PR:** the canvas `2g` note describes the digest toggle as "#97 (greenfield)" and implies the family-profile chip card doesn't exist yet. In reality, `AccountPage.tsx` already has: a "Weekly digest" `SettingsToggleRow` with cadence select (weekly/daily) around line 119-145, and full `CheckboxGroup` panels for allergies/dietary-restrictions/atmosphere-preferences (using `ALLERGEN_OPTIONS`/`DIETARY_RESTRICTION_OPTIONS`/`ATMOSPHERE_OPTIONS` from `web/src/lib/familyProfile.ts`, which already matches the backend vocab in `api/ttf_api/family_profile.py` key-for-key) around lines 172-219. This issue's job is narrower than "add a digest toggle and a profile card from scratch" — it's a **visual/component refresh**: convert the existing `CheckboxGroup` (checkbox list) rendering into the DS chip-card pattern ("WHO ARE YOU SCOUTING FOR?" eyebrow + `AttributeChip`-style pills), reusing the same `familyForm` state and `toggleKey` helper already in the file — do not rebuild the state management. If M08 is landing a shared `FamilyProfileSheet` component with chip groups, prefer wiring `AccountPage.tsx`'s "Edit" action to open that same sheet pre-filled, instead of maintaining two separate chip-rendering implementations.
6. Confirm whether `#97` (weekly digest backend — provider, templates, scheduled send) is actually implemented, or whether only the preference-toggle UI exists today. If the backend send pipeline is missing, leave the toggle as-is (it's already there) and do not claim `#97` as closed in this issue — that's tracked separately.
7. Danger zone: `AccountPage.tsx` currently renders `<SettingsPanel title="Delete account" subtitle="Permanent and cannot be undone"><DeleteAccountSettings /></SettingsPanel>` (around line 270) as a full panel. Per canvas `2g` ("one primary action" DS rule), demote this to a single text row (e.g. "Delete account & data…" in `text-error`/ttf-slow color) alongside a "Privacy · Moderation policy" line, matching the mock — move the actual deletion flow (`DeleteAccountSettings`) behind a click/confirmation rather than displaying it inline as a full panel by default.
8. Two-step verification (`MfaSettings`, line ~267) and "Signed in with Google / Sign out" rows already broadly match canvas `2g`'s "SECURITY" card — confirm styling only, no functional change expected.
