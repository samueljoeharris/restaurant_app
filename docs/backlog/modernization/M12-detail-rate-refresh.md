# Refresh restaurant detail + rate flow

Labels: next, area:web
Relates: none

## Goal
Canvas `2c`/`2d` refresh the restaurant detail page's speed display and the rate-attributes flow. Both need real code changes; one canvas claim doesn't hold up against the current API and must be scoped down or turned into a follow-up rather than shipped as claimed.

## Changes — Detail (`web/src/pages/RestaurantDetailPage.tsx`, `web/src/components/RestaurantTtfStats.tsx`, `web/src/components/ContributionRecencyChart.tsx`)
1. **Discrepancy — verify before implementing:** canvas `2c`'s spec note claims "SpeedBadge meta line now carries the daypart caveat — data the API already returns." This is **not accurate as of this audit**: `TtfAggregate` (`api/ttf_api/schemas.py`, around line 51-55) only has `sample_size`, `median_minutes`, `avg_quality`, `last_updated` — no per-daypart breakdown. `daypart` is stored per-observation (`api/ttf_api/schemas.py` line ~466/525/658, `contribution_schema.py`) and used only in admin filters (`admin.py`/`admin_moderation.py` query params), not aggregated or exposed via `RestaurantDetailResponse`. There is also no `SpeedBadge.tsx` in `web/src` — the closest equivalent is `web/src/components/RestaurantTtfStats.tsx`. Before adding "dinner runs slower (9 min)"-style copy to the detail page, either (a) add a small daypart-median aggregation to the detail endpoint (new field on `TtfAggregate` or a sibling field on `RestaurantDetailResponse`, computed from `observations` similarly to how the existing median is computed — check the aggregation code path feeding `RestaurantDetailResponse.ttf`), or (b) scope this issue down to skip the daypart caveat entirely and file it as a separate follow-up. Do not fabricate a caveat client-side from data the API doesn't provide.
2. In `web/src/components/RestaurantTtfStats.tsx`, the `Stat`/`StatGrid` rendering (around lines 30-40) is the actual "speed badge" equivalent — extend its meta line (or add one) with whatever caveat data step 1 lands, gated on sample size being large enough to split by daypart meaningfully (reuse the existing `confirmed`/`sample_size >= 3` gating pattern already in the file).
3. `web/src/components/ContributionRecencyChart.tsx` currently colors its recency bars with `bg-brand` (brand blue, hardcoded utility class around line 31 — "recent activity" bar). Per `docs/MODERNIZATION.md` recency chart should use tier colors, not brand blue. Change the bar's background to reflect the restaurant's current TTF tier (reuse `ttfTier()`/`TTF_TIER_COLORS` from `web/src/lib/ttfTier.ts`, the same helpers `RestaurantTtfStats.tsx` already imports) rather than a flat brand-colored bar for every restaurant regardless of its speed tier. This requires passing the restaurant's `ttf: TtfAggregate` into `ContributionRecencyChart` (check its current props — today it only receives `recency: ContributionRecency`) or computing tier separately in `RestaurantDetailPage.tsx` and passing a `tierColor` prop down.
4. One primary CTA: confirm `RestaurantDetailPage.tsx` currently renders one primary button ("Log a visit") plus a secondary "Saved"/watch toggle button, matching the canvas's "one primary CTA" rule — spot check only; no change expected unless a second filled button is found.

## Changes — Rate (`web/src/pages/RateAttributesPage.tsx`, `web/src/components/AttributeInput.tsx`)
5. **Verified gap:** `AttributeInput.tsx`'s `toggle` widget (around lines 21-30) currently renders exactly two `ChoiceChip`s, "Yes" and "No" — no third option. Add a "Didn't check" chip as a third value for `toggle`-type metrics, representing "no answer" distinctly from leaving the question blank (per canvas `2d`: "'Didn't check' replaces leaving blank — cleaner data than silence"). This needs:
   - A third selectable state in `AttributeInput`'s `toggle` branch (`value === true` / `value === false` / explicitly-unset-but-acknowledged) — confirm with the attribute-rating API schema (`web/src/types.ts` `MetricDefinition`/attribute submission types) whether there's already a way to submit "acknowledged, no answer" vs. simply omitting the field; if not, this may need a small API/schema change (e.g. a sentinel value) — check `api/ttf_api/routers/` attribute-submission endpoint before inventing a client-only convention that the backend can't store.
   - Apply the same "Didn't check" option to `enum_select`-type metrics too if the canvas's per-question 3-chip pattern (Yes/No/Didn't check) should generalize, or confirm it's toggle-only per the mock (canvas `2d` shows it for "High chairs?" and "Changing table?", which are toggle-type; noise/kids-menu-quality are enum_select with their own distinct options and no "didn't check" shown) — match the mock's actual scope, don't over-generalize.
6. `RateAttributesPage.tsx` submit button label: canvas `2d` wants the primary button to read "Share N ratings" where N is the count of answered questions, and a ghost "Skip for now" below it (never two filled buttons — consistent with the DS rule already followed on this page and elsewhere). Check whether this counting/label logic already exists; if the current button is a static "Submit"/"Save" label, add the dynamic count.
